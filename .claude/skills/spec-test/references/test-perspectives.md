# テスト観点リファレンス / Test Perspectives Reference

`/spec-test` パイプラインの全エージェントが従う観点集。
テストケース設計（test-designer）、仕様抽出の粒度（spec-extractor）、
検出力レビュー（mutation-verifier）の共通の判断基準とする。

## 0. 大原則 — 振る舞いを検証する

**実装の詳細ではなく、外から観測できる振る舞いを検証する。**

- 判定基準: 「実装を別の書き方に書き直してもテストが通るか？」通るなら良いテスト。
- UI は `getByRole` / `getByText` などユーザーが知覚できるものでクエリする。クラス名・DOM 構造に依存しない。
- 内部関数の呼び出し回数や順序は、それ自体が仕様（例: 課金 API は 1 回だけ呼ぶ）でない限り検証しない。

## 1. ケース設計の 4 技法（この順に適用する）

### ① 同値分割
入力を「同じ振る舞いをするグループ」に分け、各グループから代表値を 1 つ選ぶ。
最低限のグループ: 正常系 / バリデーションエラー / 認可エラー / 外部依存の失敗。

### ② 境界値分析
グループが切り替わる位置の**両側**を必ず取る。バグ（off-by-one、`<` と `<=` の取り違え）は境界に集中する。

- 数値の上下限: 上限ちょうど / 上限 + 1 / 下限ちょうど / 下限 − 1
- 長さ: 空 / 1 件 / 上限ちょうど / 上限超え
- 閾値比較: 閾値ちょうどの値と 1 つ外れた値（例: `lintEngine/titleSimilar` の編集距離 = threshold と threshold + 1）
- 境界が定数（`ACTIVITY_LIST_MAX_LIMIT` 等）の場合、テストは**定数を import して使う**（境界変更時に追従させる）

### ③ デシジョンテーブル
条件が 2 つ以上絡む場合（権限 × 公開状態、プラン × 機能フラグなど）は組み合わせ表を作り、
`it.each` のテーブルテストで実装する。

### ④ 状態遷移
状態を持つ対象は「遷移できる組み合わせ」と「**遷移できないはずの組み合わせ**」の両方を検証する。

## 2. 見落としやすい観点チェックリスト

- [ ] **空・ゼロ・null**: 空配列、空文字、0 件ヒット、undefined
- [ ] **認可**（認証とは別）: 他人のリソースへのアクセスが 403/404 になるか。401 だけで満足しない
- [ ] **エラー後の状態**: 外部依存が失敗したとき、フォールバック・通知・ロールバックが仕様通りか
- [ ] **冪等性・重複実行**: 同じ操作を 2 回実行したら？（保存ボタン連打、Webhook の再送）
- [ ] **時間**: タイムゾーン、月末・年末。`vi.useFakeTimers` / `vi.setSystemTime` で固定する
- [ ] **文字**: マルチバイト、絵文字、サロゲートペア（文字数カウント・切り詰め・正規化処理で頻出）
- [ ] **並行・競合**: 共同編集（Yjs）、キャッシュ無効化、デバウンス/スロットルが絡む箇所

## 3. テストの構造

- **AAA（Arrange-Act-Assert）** を空行で区切る。準備が長い場合はビルダー/ヘルパーへ
  （既存の `createMockDb`、`createTestApp` のパターンに従う）。
- **テスト名は仕様の文章**: `it("上限 200 を超える limit は 200 に丸める")`。
  失敗時にテスト名だけで何が壊れたか分かること。
- **1 テスト 1 関心事**。アサーションは複数でよいが、検証する振る舞いは 1 つ。
- **テスト内のロジック禁止**: if / for / 期待値の計算を書かない。期待値はハードコードする。
- describe はコンポーネント・関数単位。コメントは日英併記（AGENTS.md）。

## 4. 信頼性 — 嘘をつかないテスト

- **決定的にする**: `Date.now()` / `Math.random()` は固定、ネットワークはモック。
  `waitForTimeout` ではなく条件待ち（`await expect(...).toBeVisible()` / `waitFor`）。
- **独立にする**: テスト間で状態を共有しない。実行順序を変えても通ること。
- **一度 fail を確認する**: 期待値を意図的に壊して落ちることを確認してから戻す。
  「常に通るテスト」が最悪のテスト。

## 5. Mutation 耐性のあるアサーション

Mutation スコアが品質の第一指標（AGENTS.md）。以下は生存ミュータントの典型原因:

| 弱いアサーション | 強いアサーション |
| --- | --- |
| `expect(fn).toHaveBeenCalled()` | `expect(fn).toHaveBeenCalledWith(具体値)` |
| `expect(result).toBeTruthy()` | `expect(result).toEqual({ ... })` |
| `expect(items.length).toBeGreaterThan(0)` | `expect(items).toHaveLength(3)` |
| エラーが「投げられること」のみ | エラーメッセージ・コードまで検証 |
| 境界の片側のみ | 境界の両側（§1-②） |

## 6. モックの規律

- モックしてよいのは**自分が所有しない境界**のみ: HTTP、DB、時計、ファイルシステム、外部 SDK。
- 自分のコード同士の呼び出しをモックしない（実装の写し絵になり、リファクタで壊れて何も守らない）。
- モックの応答は**実際の API が返す形**のフィクスチャを使う。都合のよい形を発明しない。

## 7. レイヤーごとの責務分担

| レイヤー | 守るもの | 書く量 |
| --- | --- | --- |
| 単体（lib / hooks / services） | ロジックの正しさ、境界値、エラー分岐の網羅 | 最多 |
| コンポーネント / ルート統合 | 配線: 状態→表示（loading / error / empty / data）、リクエスト→レスポンス、認可 | 状態ごとに代表ケース |
| E2E（Playwright） | クリティカルジャーニーの一筆書き | 最少。分岐を持ち込まない |

**同じ観点を複数レイヤーで重複させない。** バリデーションの全境界は単体で網羅し、
ルート統合では「400 が返る」代表 1 ケース、E2E では触れない。

## 8. アンチパターン（レビュー時のチェック用）

- カバレッジを上げるためだけのテスト（アサーションが薄い、スナップショット偏重）
- 巨大な beforeEach に全準備を詰め、各テストの前提が読めない
- プライベート関数を export してテストする（公開 API 経由で検証するか、独立モジュールに切り出す）
- 落ちたテストを `skip` で塞ぐ（直すか削除する）
- テストのためだけの本体コード変更（テスト用フック、`isTest` 分岐）
- **実装に合わせて期待値を調整する**（本パイプラインが構造的に防いでいる対象。絶対にしない）
