# Chrome拡張 Phase 2 認証方式の比較

**目的**: ワンクリック保存のための拡張認証方式と専用API設計案を比較する。

---

## 背景

Phase 1 MVP では、拡張から Web アプリに `clipUrl` を渡し、アプリ内で認証・保存を行う方式を採用した。
これにより拡張から直接 API を叩かず、Cookie 認証・CSRF 制約の問題を回避している。

Phase 2 では、拡張から**直接**「取得→抽出→ページ作成」までを完結させる「真のワンクリック保存」を検討する。

---

## 認証方式の比較

| 方式                                   | 概要                                                                                                                                        | メリット                                                     | デメリット                                                                                          |
| -------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------ | --------------------------------------------------------------------------------------------------- |
| **A. OAuth 2.0 + PKCE**                | 拡張がブラウザタブで OAuth 認証を行い、authorization code を PKCE で交換して access token を取得。拡張専用 API では Bearer トークンで認証。 | 標準的な OAuth フロー、既存 IdP（Google/GitHub）と統合可能。 | 初回ログイン時にタブを開く必要がある。拡張の秘密管理（client_secret 不要だが state 管理等）が必要。 |
| **B. 専用拡張トークン**                | Web アプリの「設定」で「Chrome拡張用トークン」を発行。ユーザーが手動で拡張オプションに貼り付け。                                            | 実装がシンプル。既存認証フローに影響しない。                 | ユーザー操作が 2 ステップ必要。トークン漏洩時の取り扱い（無効化、有効期限）を自前で実装する必要。   |
| **C. Cookie + externally_connectable** | 拡張に `externally_connectable` で Web アプリからの `postMessage` を許可し、アプリが取得した Cookie を拡張に渡す。                          | 既存の Cookie 認証をそのまま利用可能。                       | セキュリティリスクが高い。Cookie を拡張に渡す経路の設計が複雑。攻撃面が増える。                     |

## API 設計案

### 拡張専用エンドポイント候補

| エンドポイント                  | 方法                                                              | 用途                 |
| ------------------------------- | ----------------------------------------------------------------- | -------------------- |
| `POST /api/ext/session`         | 拡張トークン発行（OAuth code または設定画面トークン）             | 初回 or リフレッシュ |
| `POST /api/ext/clip-and-create` | URL 受領 → サーバー側で HTML 取得 → Readability 抽出 → ページ作成 | ワンクリック保存     |

### clip-and-create の想定シグネチャ

- **Request**: `{ url: string }`
- **Response**: `{ page_id: string, title: string, thumbnail_url?: string, error?: string }`
- **認証**: `Authorization: Bearer <extension_token>`

---

## 正式決定（UX 最優先）

**採択方式**: **A. OAuth 2.0 + PKCE**

UX を最優先する方針として、OAuth 2.0 + PKCE を正式な拡張認証方式として決定する。

### 決定理由

- 初回セットアップが 1 回で完結し、手動コピペ（B）が不要
- 以後はポップアップを開かずにワンクリック保存できる継続利用 UX が最良
- Cookie 受け渡し（C）のような危険設計を避けられ、将来機能追加でも再設計が少ない

### 技術要件

- 拡張は `chrome.identity.launchWebAuthFlow` を使用する（manifest に `identity` 権限を追加）
- 短命アクセストークン + リフレッシュトークン（またはサイレント再認可）で、バックグラウンドで透過更新する

### 従来推奨（コスト優先時）

- **B（専用拡張トークン）** は実装コストが低い妥協案として依然有効。短期リリース向け。

---

## API 設計（正式）

| エンドポイント                  | 方法                                                              | 用途                   |
| ------------------------------- | ----------------------------------------------------------------- | ---------------------- |
| `POST /api/ext/session`         | OAuth code + PKCE でトークン発行・リフレッシュ                    | 初回認証・トークン更新 |
| `POST /api/ext/clip-and-create` | URL 受領 → サーバー側で HTML 取得 → Readability 抽出 → ページ作成 | ワンクリック保存       |

認証は `Authorization: Bearer <extension_token>` を全エンドポイントで使用する。
