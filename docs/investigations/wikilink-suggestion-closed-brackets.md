# WikiLink 入力補助: 閉じた `]]` 内での発動抑制

## 概要

`[[` 入力時に表示される WikiLink の入力補助（サジェスト）が、`]]` で完全に囲まれたリンク内にカーソルがある場合にも発動してしまう問題。かっこが完全に閉じているときは入力補助を発動させないようにしたい。

## 現状の実装

### 関連ファイル

| ファイル                                                       | 役割                                  |
| -------------------------------------------------------------- | ------------------------------------- |
| `src/components/editor/extensions/wikiLinkSuggestionPlugin.ts` | `[[` パターン検知とサジェスト状態管理 |
| `src/components/editor/extensions/WikiLinkSuggestion.tsx`      | サジェスト UI                         |
| `src/components/editor/TiptapEditor/useSuggestionEffects.ts`   | サジェスト選択時の挿入処理            |

### 判定ロジック（`wikiLinkSuggestionPlugin.ts`）

```typescript
// カーソル直前のテキストを取得
const textBefore = $from.parent.textBetween(0, $from.parentOffset, null, "\ufffc");

// [[ を含み、最後の ]] 以降の [[... パターンを検出
const match = textBefore.match(/\[\[([^\]]*?)$/);

if (match) {
  // サジェストを active にする
}
```

- **検知対象**: カーソル直前のテキストで `[[` + `]` を含まない文字列 + 末尾、というパターン
- **問題**: カーソルが `[[Page]]` 内（例: "Page" の途中）にある場合、`textBefore` は `[[Pa` のようになり、正規表現にマッチしてしまう
- **不足している判定**: 対応する `]]` がカーソルより後ろにあるかどうかのチェック

### 発動してしまうケース

| 内容          | カーソル位置     | textBefore  | 結果                                              |
| ------------- | ---------------- | ----------- | ------------------------------------------------- |
| `[[Page]]`    | "Page" の P の後 | `[[P`       | マッチ → 発動してしまう                           |
| `[[Page]]`    | "Page" の e の後 | `[[Page`    | マッチ → 発動してしまう                           |
| `[[A]] [[B]]` | "B" の途中       | `[[A]] [[B` | マッチ → 発動してしまう（この例は意図通りでよい） |

上記 1, 2 は、`]]` で閉じているため発動させたくないケース。

### 発動してほしいケース（現状も問題なし）

| 内容                | カーソル位置 | textBefore          | 結果                |
| ------------------- | ------------ | ------------------- | ------------------- |
| `[[Page`            | 末尾         | `[[Page`            | マッチ → 発動（OK） |
| `[[Link1]] [[Link2` | Link2 の末尾 | `[[Link1]] [[Link2` | マッチ → 発動（OK） |
| `see [[P`           | 末尾         | `see [[P`           | マッチ → 発動（OK） |

## 想定ユースケース

### 1. リンク内のタイトル編集

- ユーザーが `[[既存ページ]]` をクリックし、タイトルを編集したい
- カーソルが "既存ページ" 内に入る
- このときサジェストが出ると、編集がしづらい

### 2. リンクテキストのコピー・貼り付け

- `[[Page A]]` 内でテキストを選択してコピーし、別の `[[Page B]]` 内に貼り付け
- 貼り付け後もカーソルは `[[...]]` 内にある
- サジェストが出ると作業の邪魔になる

### 3. リンクを Backspace で部分的に削除

- `[[Page]]` 内で Backspace を押して "ge" を消す → `[[Pa` の状態
- このときは `]]` が消えているため、サジェストは発動してよい

### 4. 複数リンクの連続入力

- `[[A]] [[B]] [[C` のように続けて入力
- `[[C` の時点では未閉じなのでサジェスト発動は正しい

## 修正方針

カーソル以降のテキストで `]]` の有無を確認し、マッチした `[[` がすでに `]]` で閉じているかどうかを見る。

```typescript
const textAfter = $from.parent.textBetween(
  $from.parentOffset,
  $from.parent.content.size,
  null,
  "\ufffc",
);

// カーソル以降に ]] が、新しい [[ より前にあれば「閉じたリンク内」
const idxClose = textAfter.indexOf("]]");
const idxOpen = textAfter.indexOf("[[");
const isInsideClosedLink = idxClose >= 0 && (idxOpen < 0 || idxClose < idxOpen);

if (match && isInsideClosedLink) {
  // サジェストを発動させない
  if (prev.active) {
    /* deactivate */
  }
  return prev;
}
```

- **利点**: プレーンテキスト `[[...]]`（まだ wikiLink マークが付いていない段階）にも対応可能
- **注意**: 発動するのは `idxClose === -1`（カーソル以降に `]]` なし）のとき、または `idxOpen >= 0` かつ `idxClose >= idxOpen`（`[[` が `]]` より前に出現）のとき。同じ行内に `[[A]]` と `[[B` がある場合、`[[B` 側では `idxClose === -1` のため発動する（想定通り）

## 懸念事項・検討すべき点

### 1. ブロック境界をまたぐ場合

- `$from.parent` は直親ノード（段落など）のみを対象とする
- ブロックをまたぐ `[[...]]` は通常存在しないため、実運用上の問題は小さい想定

### 2. codeBlock / インラインコード内の `[[...]]`

- `contentUtils` では codeBlock 内の `[[1, 2]]` 等は wikiLink に変換しない
- 同様に、サジェストも「入力補助が出てよい文脈」に限定する必要があるかは要検討
- 現状は親の `textBetween` で判定しているだけなので、codeBlock 内でも `[[` があれば発動する。仕様として許容するか、コードブロック内では無効にするかを決める必要がある

### 3. ネストされた `[[` の扱い

- 現状の正規表現は「最後の `[[` から `]` を含まない範囲」をクエリとしている
- `[[A|[[B]]]]` のような Obsidian 風記法は想定していないようなので、当面は現状のロジックで十分と考えられる

### 4. 既存 E2E テスト

- `e2e/linked-pages.spec.ts` で `[[` によるサジェストを検証している
- 修正後は「閉じたリンク内では発動しない」ケースを追加して regress を防ぐとよい

### 5. パフォーマンス

- `textBetween` は既に呼んでいるため、`textAfter` の取得による追加コストは小さい
