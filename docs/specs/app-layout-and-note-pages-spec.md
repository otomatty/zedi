# アプリレイアウト・ノートページ仕様書

**バージョン**: 1.0  
**対象実装**: 共通レイアウト（サイドバー・AI チャットドック）およびノート関連ページの構成・振る舞い  
**作成日**: 2025-03-19

---

## 1. 概要

### 1.1 目的

本仕様は、アプリ全体で共通するレイアウト構造（ヘッダー・左サイドバー・メインコンテンツ・右 AI チャットドック）および、ノートの表示・設定・メンバー管理ページの機能・権限・UI 構成を定義する。実装の一貫性と今後の拡張の基準とする。

### 1.2 スコープ

- **対象**: 共通レイアウト、ヘッダー・サイドバー・AI チャットドック、ノートビュー・ノート設定・ノートメンバー画面、ページ一覧グリッドの列数制御、AI チャットの表示制御（レイアウト層 vs ページ局所）。
- **対象外**: 認証・API・バックエンド、エディタ内部、管理画面、E2E テスト手順の詳細。

---

## 2. 共通レイアウトアーキテクチャ

### 2.1 レイアウト構造

アプリの主要画面は、次の 4 領域で構成される。

| 領域                 | 説明                                                      | 備考                                                                                           |
| -------------------- | --------------------------------------------------------- | ---------------------------------------------------------------------------------------------- |
| ヘッダー             | 全幅・画面上部固定。高さは CSS 変数で定義（例: 4.5rem）。 | サイドバートリガー、ロゴ、検索、AI チャットボタン、ユニファイドメニューを含む。                |
| 左サイドバー         | ヘッダー直下から画面下端まで。開閉可能。                  | デフォルトは閉。開閉状態はプロバイダで管理。                                                   |
| メインコンテンツ     | ヘッダー直下、サイドバー右側。可変幅。                    | 縦スクロール可能。最小幅 0 で縮小を許容。                                                      |
| 右 AI チャットドック | ヘッダー直下から画面下端まで。開閉可能。                  | 表示幅は CSS 変数で定義（例: 22rem）。メインコンテンツはドック表示時にその分だけ左に押される。 |

レイアウトは「ヘッダー全幅 ＋ その下で 左サイドバー | メイン | 右ドック」の 3 カラム構造とする。ヘッダーはサイドバーより上にあり、サイドバーはヘッダーの高さを除いた高さで表示する。

### 2.2 CSS 変数（レイアウト用）

レイアウトプロバイダで次の変数が設定される。

| 変数名                | 用途                                                               | 代表値 |
| --------------------- | ------------------------------------------------------------------ | ------ |
| `--app-header-height` | ヘッダー高さ。サイドバー・AI ドックの top および高さ計算に使用。   | 4.5rem |
| `--ai-chat-width`     | デスクトップ時の AI チャットパネル幅。スペーサー・パネル幅に使用。 | 22rem  |

サイドバーおよび AI ドックの高さは `calc(100svh - var(--app-header-height))` で統一する。

### 2.3 適用画面

次の画面は、いずれもこの共通レイアウトでラップされる。

- ホーム（ページ一覧）
- ノート一覧
- 検索結果
- ノート詳細（ノートビュー）
- ノート設定
- ノートメンバー
- ノート内ページ表示（ノートページビュー）

## 3. ヘッダー

### 3.1 構成

- **左**: サイドバー開閉トリガー（アイコン＋アクセシブルなラベル）。その右に、デスクトップ時のみロゴと月ナビゲーションを表示。モバイルではトリガーのみとし、ロゴ・月ナビは非表示。
- **中央**: グローバル検索コンテキストがある場合に検索バーを表示。AI チャット開閉ボタン。
- **右**: 未サインイン時はゲスト向け案内（デスクトップのみ）、サインイン状態に応じたユニファイドメニュー。

ヘッダーはスティッキーで、背景は半透明＋ブラーを想定。

### 3.2 サイドバートリガー

- クリックで左サイドバーの開閉を切り替える。
- アクセシビリティのため、メニューを開くボタンであることが分かるラベル（例: 「メニュー」）を付与する。

---

## 4. 左サイドバー

### 4.1 基本仕様

- **配置**: 左側。ヘッダー下から画面下端まで。
- **開閉**: プロバイダのデフォルトは「閉」。開閉状態はプロバイダで保持し、トリガーで切り替え。
- **レスポンシブ**: モバイルではオフキャンバス（例: シート／ドロワー）で表示。デスクトップではインラインで開閉。

### 4.2 ナビゲーション項目

| 表示ラベル（i18n キー想定） | パス      | 備考                                    |
| --------------------------- | --------- | --------------------------------------- |
| Home                        | /home     | 厳密一致でアクティブ。                  |
| Notes                       | /notes    | パスが /notes で始まるとアクティブ。    |
| Settings                    | /settings | パスが /settings で始まるとアクティブ。 |
| Plan                        | /pricing  | パスが /pricing で始まるとアクティブ。  |

各項目はアイコン＋テキストで表示し、アクティブな項目は視覚的に強調する。ツールチップで同じラベルを表示する。

### 4.3 ヘッダー・フッター

- **サイドバーヘッダー**: アプリ名（例: Zedi）を表示し、ホームへのリンクとする。フォーカス時はリングで視覚フィードバック。
- **サイドバーフッター**: サインイン時のみ表示。ユーザーアバター（画像またはイニシャル）と表示名（プロフィールまたはユーザー情報から取得、取得できない場合は「—」等）を表示。クリックやナビゲーションは本仕様では規定しない。

## 5. AI チャットドック（レイアウト層）

### 5.1 表示条件

- アプリ全体で「AI チャットが利用可能」とマークされている場合のみ、レイアウト層の AI チャットドックを描画する。
- 利用可能フラグは、AI チャットを使う画面がマウントされている間 true、アンマウント時に false にし、使用しない画面ではドックを出さない。

### 5.2 デスクトップ

- メインコンテンツの右端に固定幅のパネルとして表示する。
- 幅は `--ai-chat-width` に従う。開閉時は幅・ボーダー・不透明度をトランジション（例: 300ms）で切り替える。
- 閉じている間は幅 0・クリック無効とし、メインコンテンツがパネルに重ならないようにする。
- レイアウトのフロー上には、開いているときのみ幅 `--ai-chat-width` のスペーサーを置き、メインコンテンツがパネルの下に隠れないようにする。

### 5.3 モバイル

- ドロワー（下または横から出るオーバーレイ）で表示する。
- ドロワー内の高さは画面の一定割合（例: 85vh）とする。
- 開閉状態はグローバルな AI チャット状態と同期し、閉じる操作でドックを閉じる。

### 5.4 状態の永続化

- AI チャットの開閉状態およびコンテキスト有効フラグは永続化する。
- 永続化ストレージにはバージョン番号（例: 1）を持たせ、将来のスキーマ変更に備える。

---

## 6. AI チャットの「レイアウト層」と「ページ局所」の使い分け

### 6.1 二つのモード

- **レイアウト層（グローバルドック）**: 共通レイアウトに組み込まれた右側の AI チャットドック。ホーム・ノート一覧・検索結果・ノートビュー・ノート設定・ノートメンバー・ノートページビューなど、共通レイアウトを使う画面では、AI チャットはこのドックにのみ表示する。ページ側では自前の AI パネルを描画しない。
- **ページ局所（ローカルパネル）**: 共通レイアウトを使わない、またはレイアウトと独立した全画面に近い画面（例: ページエディタ）では、その画面内で AI チャットパネル（またはモバイルではドロワー）を描画する。このとき、レイアウト層のドックは描画しない（AI 利用可能フラグを出さない）。

### 6.2 フローティングアクションボタン（FAB）

- コンテンツラッパーが FAB を表示する場合、その位置は「レイアウト層のドックが開いているか」「ページ局所でパネルを出しているか」に応じて右端をずらす。
- レイアウト層のみ使用時: ドックが開いていれば、FAB はドック幅分だけ左に寄せる。ページ局所使用時: その画面内でパネルが開いていれば同様にオフセットする。
- セーフエリア（例: safe-area-inset-bottom, safe-area-inset-right）を考慮した余白を付与する。

### 6.3 ページドラッグ時のヒント

- AI チャットが閉じている状態で、ページカード等をドラッグしているとき、一定時間特定ゾーンにホバーすると AI チャットを開くヒントを出す仕様は、レイアウト層・ページ局所のどちらでパネルを描画する場合でも同様に扱う（該当画面で実装する場合）。

## 7. ノートビュー（ノート詳細）

### 7.1 役割

- 1 つのノートを識別し、そのノートのタイトル・公開範囲バッジ・公式バッジ・ページ数、およびノートに含まれるページの一覧を表示する。
- 権限に応じて「ノートにページを追加」「ページを削除」「メンバー管理へのリンク」等を提供する。

### 7.2 権限の考え方

- ノートのアクセス情報（閲覧・編集・ページ追加・メンバー管理）と、ノートが「ローカル」か「リモート」かを組み合わせて、画面で使う権限フラグを算出する。
- **編集**: アクセスで編集可かつノートがローカルである場合にのみ true。
- **ページ追加**: アクセスでページ追加可であれば true（ローカル限定ではない）。
- **「ページを追加」UI の表示**: 編集可またはページ追加可のどちらかであれば表示する。
- **メンバー管理**: アクセスでメンバー管理可かつノートがローカルである場合にのみ true（メンバー管理リンク・設定はローカルノートのみ）。

### 7.3 画面構成

- **読み込み中**: 共通レイアウトのまま、ローディングメッセージを表示。
- **存在しない／アクセス不可**: 共通レイアウトのまま、該当メッセージを表示。
- **正常**:
  - 見出し: ノートタイトル（未設定時は「無題のノート」等）、公開範囲バッジ、公式バッジ（該当時）、ページ数。
  - ヘッダーアクション: メンバー管理へのリンク（メンバー管理可の場合）、「ページを追加」ボタンおよび追加用ダイアログ（ページを追加可能な場合）。ダイアログでは「新規タイトルで追加」と「既存ページから選択」の両方に対応する。
  - メイン: ノートに含まれるページのグリッド。各カードからノートから削除する操作（権限がある場合）。ページが 0 件のときは専用メッセージを表示。

### 7.4 データ

- ノート情報・アクセス・ノートに含まれるページ一覧・全ページサマリーは既存の API／フックで取得する。ノートに含まれるページは「追加者」情報を持つ拡張サマリーとして扱う。

## 8. ノート設定

### 8.1 役割

- ノートの設定（タイトル・公開範囲・編集権限）の表示・編集、共有リンクのコピー、ノート削除を行う。

### 8.2 権限

- ノートの閲覧権限がない、またはノートが取得できない場合は、共通レイアウトのままエラーメッセージを表示。
- 「メンバーを管理できる」かつ「ノートがローカル」の場合のみ、設定の編集（タイトル・公開範囲・編集権限の保存、削除）を許可する。それ以外の場合は設定フォームを表示せず、権限がない旨のメッセージを表示する。

### 8.3 画面構成

- **共有リンク**: ノートの URL を読み取り専用で表示し、コピーボタンでクリップボードにコピーする。成功・失敗はトーストで通知。
- **公開範囲・編集権限**: 公開範囲（例: 非公開・公開・未掲載・制限付き）と編集権限（例: オーナーのみ・メンバー編集可・ログイン済み誰でも）を選択する。公開範囲に応じて選択可能な編集権限の組み合わせを制限する。保存ボタンで API に送信し、結果をトーストで通知。
- **削除**: 危険ゾーンとして削除ボタンを配置。クリックで確認ダイアログを表示し、確認後に削除を実行。成功時はノート削除後にホーム等へ遷移する。

### 8.4 公開範囲と編集権限の対応

- 非公開: 編集権限は「オーナーのみ」のみ。
- 制限付き: 「オーナーのみ」「メンバー編集可」。
- 未掲載・公開: 「オーナーのみ」「メンバー編集可」「ログイン済み誰でも」。
- 公開範囲変更時に、現在の編集権限が新しい公開範囲で許容されない場合は、その公開範囲で最初に許容される編集権限に自動で切り替える。

## 9. ノートメンバー

### 9.1 役割

- ノートのメンバー一覧の表示、メンバー追加・ロール変更・削除を行う。

### 9.2 権限

- ノートの閲覧権限がない、またはノートが取得できない場合は、共通レイアウトのままエラーメッセージを表示。
- 「メンバーを管理できる」かつ「ノートがローカル」の場合のみ、メンバー管理 UI（追加・ロール変更・削除）を表示する。それ以外の場合は管理 UI を出さず、権限がない旨のメッセージを表示する。

### 9.3 画面構成

- 見出し: 「メンバー」とノートタイトル。ノートへ戻るリンク。
- **管理可能時**:
  - メンバー追加: メールアドレス入力、ロール選択（例: 閲覧者・編集者）、追加ボタン。追加成功・失敗はトーストで通知。
  - メンバー一覧: 各メンバーのメールアドレス、ロール選択、削除ボタン。ロール変更・削除の結果はトーストで通知。
- **管理不可時**: メンバー管理権限がない旨のメッセージを表示。

## 10. ページ一覧グリッドの列数

### 10.1 要件

- ページカードのグリッドは、**ビューポート**ではなく**コンテナの幅**に応じて列数（2〜6）を変える。
- これにより、左サイドバーや右 AI ドックの開閉でコンテナ幅が変わったときも、表示列数が適切に変わる。

### 10.2 列数の算出

- コンテナの幅を ResizeObserver で監視する。
- 幅に応じた閾値で列数を決定する（例: 360px 未満→2、520px 未満→3、680px 未満→4、880px 未満→5、以上→6）。閾値はカードをやや小さくして多めの列数になるよう調整する。
- 初回描画時はレイアウト効果で即時に計測し、スケルトンからコンテンツに切り替わる際も列数がずれないようにする。

### 10.3 スケルトン

- 読み込み中はスケルトンのグリッドを表示する。そのグリッドの列数は、上記と同じ「コンテナ幅から算出した列数」を使い、本コンテンツ表示時と一致させる。

## 11. 国際化（i18n）

- ヘッダー・サイドバー・ノート関連のラベル・メッセージはすべて翻訳キー経由で表示する。
- ナビゲーションでは、Home / Notes / Settings / Plan / Menu 等のキーを使用する。
- ノート設定では公開範囲・編集権限のラベル、ノートメンバーではロール名（閲覧者・編集者等）、共通で「保存」「読み込み中」「コピー」等のキーを使用する。キーが未定義の場合はフォールバック文言（例: 英語）を表示する。

---

## 12. アクセシビリティ

- サイドバートリガーにはメニューであることが分かるラベルを付与する。
- フォーカス可能要素にはフォーカスリング等の視覚的フィードバックを付与する。
- AI ドックのスペーサーなど、装飾的でクリック対象でない要素には `aria-hidden` を付与する。
- フォーム（ノート設定・メンバー追加等）ではラベルと入力の紐付け、破壊的操作には確認ダイアログを用意する。
