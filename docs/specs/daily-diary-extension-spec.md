## 1. 概要

### 1.1 目的

Zedi の拡張機能として**日記自動作成**機能を提供する。毎日自動的にその日の日記ページを作成し、ユーザーが日記を書く習慣を支援する。

### 1.2 ユーザーストーリー

> 👤 **ユーザー A（日常的なメモ書きユーザー）**
>
> 「毎日日記を書きたいが、ページを手動で作成するのが面倒。アプリを開いたら今日の日記ページがすでに用意されていてほしい。」

> 👤 **ユーザー B（振り返り重視のユーザー）**
>
> 「日記にはその日の天気、気分、TODO リストなど決まったフォーマットがある。テンプレートから自動生成されると助かる。」

> 👤 **ユーザー C（ワークログを残すユーザー）**
>
> 「業務日報として使いたい。今日の作業内容、明日の予定、メモ欄があるテンプレートで毎朝自動生成してほしい。」

### 1.3 設計原則

| 原則 | 説明 |
|------|------|
| **ゼロフリクション** | ユーザーの操作なしに日記ページが準備される |
| **カスタマイズ可能** | テンプレート・タイトル形式・作成時刻を設定可能 |
| **既存機能との統合** | WikiLink・Date Grid・AI 機能と自然に連携 |
| **重複防止** | 同じ日のページが二重に作成されない |

## 2. 機能要件

### 2.1 コア機能

| # | 機能 | 説明 | 優先度 |
|---|------|------|--------|
| F1 | 日記ページ自動作成 | 指定時刻に今日の日記ページを自動作成 | 高 |
| F2 | テンプレート設定 | 日記のフォーマットをテンプレートで定義 | 高 |
| F3 | タイトル形式設定 | 日記ページのタイトル形式をカスタマイズ | 中 |
| F4 | 重複チェック | 同日のページが既に存在する場合はスキップ | 高 |
| F5 | 手動作成 | `/diary` スラッシュコマンドで手動作成 | 高 |
| F6 | アプリ起動時チェック | アプリ起動時に今日の日記がなければ作成 | 高 |
| F7 | 日記一覧表示 | 日記ページを時系列で一覧表示 | 中 |
| F8 | 通知 | 日記作成時にトースト通知を表示 | 低 |

### 2.2 テンプレートシステム

#### 2.2.1 デフォルトテンプレート

```markdown
# {{date}} の日記

## 今日の気分
- 

## 今日やったこと
- 

## 明日やること
- 

## メモ

```

#### 2.2.2 テンプレート変数

| 変数 | 説明 | 出力例 |
|------|------|--------|
| `{{date}}` | 日付（ロケール依存） | `2026年3月8日` / `March 8, 2026` |
| `{{date_short}}` | 短縮日付 | `2026/03/08` |
| `{{date_iso}}` | ISO 8601 日付 | `2026-03-08` |
| `{{weekday}}` | 曜日 | `月曜日` / `Monday` |
| `{{year}}` | 年 | `2026` |
| `{{month}}` | 月 | `3` |
| `{{day}}` | 日 | `8` |

#### 2.2.3 カスタムテンプレート

ユーザーは既存のページをテンプレートとして指定可能。テンプレートページの内容がコピーされ、テンプレート変数が展開される。

### 2.3 タイトル形式

設定可能なタイトルパターン：

| パターン | 出力例 |
|----------|--------|
| `{{date_iso}}` | `2026-03-08` |
| `日記 {{date}}` | `日記 2026年3月8日` |
| `{{date}} ({{weekday}})` | `2026年3月8日 (日曜日)` |
| `Daily Note {{date_short}}` | `Daily Note 2026/03/08` |
| カスタム | ユーザー定義のパターン |

### 2.4 スラッシュコマンド

`/diary` コマンドで今日の日記ページを手動作成：

```
ユーザーが / を入力
  → スラッシュコマンドメニューが表示
  → 📅 日記を作成 を選択
  → 今日の日記ページが作成される（または既存ページに遷移）
```

### 3.4 既存機能との連携

#### Date Grid との連携

日記ページは Date Grid（日付別ノート一覧）に自然に表示される。日記ページに特別なメタデータ（`isDiary: true`）を付与し、Date Grid で視覚的に区別可能にする。

#### WikiLink との連携

日記テンプレートに `[[プロジェクト名]]` などの WikiLink を含めることで、日記から関連ページへの自動リンクを生成。

#### AI 機能との連携

将来的に AI が前日の日記内容を要約し、「昨日のハイライト」セクションを自動生成する機能も検討。

## 4. 要件定義のための質問・提案

### 4.1 質問事項

| # | 質問 | 選択肢 / 補足 |
|---|------|--------------|
| Q1 | 日記の自動作成はサーバーサイド・クライアントサイドどちらが望ましいか？ | **A)** サーバーサイド（cron ジョブ）— ブラウザ未起動でも動作<br>**B)** クライアントサイド（アプリ起動時）— サーバー負荷なし<br>**C)** 両方（サーバーで作成 + クライアントで補完） |
| Q2 | 日記ページは通常のページと同じデータ構造で良いか？ | 専用のフィールド（`isDiary`, `diaryDate`）を追加するか |
| Q3 | 過去の日記を遡って作成する機能は必要か？ | 例：旅行から戻った後にまとめて日記を作成 |
| Q4 | 日記テンプレートのエディターは必要か？ | Tiptap エディターでテンプレートを WYSIWYG 編集するか、Markdown テキストで定義するか |
| Q5 | 日記のタグ自動付与は必要か？ | 例：`#diary`, `#2026-03` のようなタグの自動付与 |
| Q6 | 複数の日記シリーズをサポートするか？ | 例：「個人日記」と「業務日報」を別テンプレートで |
| Q7 | 日記の共有・公開機能は将来的に必要か？ | 日報を同僚と共有するユースケース |

### 4.2 提案事項

#### 提案 1: MVP はクライアントサイドのみで実装

**初期実装ではアプリ起動時（+ `/diary` コマンド）での作成のみとし、サーバーサイドスケジューラーは Phase 2 以降で実装する。**

理由：
- サーバーサイドのスケジューラー実装は複雑（タイムゾーン管理、エラーハンドリング等）
- ユーザーがアプリを毎日使う前提であれば、起動時チェックで十分
- 既存の `PageRepository` をそのまま活用できる
- フィーチャーフラグで段階的にリリース可能

#### 提案 2: 既存のページ構造を活用

**日記ページは通常のページとして作成し、タイトルの命名規則（例: `2026-03-08`）で識別する。**

理由：
- データベーススキーマの変更が不要
- 既存の検索・WikiLink・Date Grid がそのまま使える
- ページのメタデータ（将来的な `tags` フィールド）で `diary` タグを付ければ十分
- シンプルな実装で素早くリリース可能

#### 提案 3: テンプレートは Tiptap JSON で保存

**テンプレートは Tiptap の JSON 形式で保存し、Tiptap エディターで編集可能にする。**

理由：
- 既存のエディターをそのまま活用できる
- リッチテキスト（見出し、リスト、チェックボックス等）をテンプレートに含められる
- Markdown 変換のオーバーヘッドがない
- テンプレート変数は `{{variable}}` をテキストノード内に含める形で対応
