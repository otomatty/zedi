# 仕様書: Chrome 拡張 UX 原則検証

**日時**: 2026-03-15  
**対象**: Chrome 拡張 Phase 2 認証・API 設計・将来機能計画

---

## 1. UX 原則（検証基準）

| #   | 原則                 | 説明                                                                                                                                 |
| --- | -------------------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| P1  | **ワンクリック保存** | ユーザーが拡張アイコンまたはショートカットを 1 回押すだけで、現在のページが zedi に保存される。追加の入力・選択は不要。              |
| P2  | **再認証透過**       | トークン期限切れ時はバックグラウンドで自動更新し、ユーザーに再ログインを求めない。ユーザーは認証状態を意識しない。                   |
| P3  | **最小操作**         | 初回セットアップ後、日常利用で必要な操作は最小限。ポップアップを開く、フォーム入力する、手動コピペするなどの余計なステップを避ける。 |

---

## 2. 認証方式（OAuth 2.0 + PKCE）の検証

| 原則   | 検証結果 | 詳細                                                                                                                                                             |
| ------ | -------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **P1** | ✅ 適合  | 初回認証後は `Authorization: Bearer` で API を呼び、ポップアップなしで保存可能。                                                                                 |
| **P2** | ✅ 適合  | 短命トークン + リフレッシュトークン（または `launchWebAuthFlow` によるサイレント再認可）で、バックグラウンドで透過更新。ユーザーに再ログイン画面を出さない設計。 |
| **P3** | ✅ 適合  | 初回は 1 回だけ OAuth 認可画面（タブ）で「許可」を押すだけで完結。設定画面でのトークンコピペ（方式 B）は不要。                                                   |

**代替案との比較**

| 方式                | P1  | P2     | P3                                   |
| ------------------- | --- | ------ | ------------------------------------ |
| A. OAuth + PKCE     | ✅  | ✅     | ✅ 初回 1 クリック                   |
| B. 専用拡張トークン | ✅  | 要実装 | ❌ 設定→コピー→貼り付けの 3 ステップ |
| C. Cookie 受け渡し  | ✅  | 複雑   | △ 初回は複雑                         |

## 3. API 設計（capability スコープ）の検証

| 原則   | 検証結果 | 詳細                                                                                                               |
| ------ | -------- | ------------------------------------------------------------------------------------------------------------------ |
| **P1** | ✅ 適合  | `clip:create` スコープで `POST /api/ext/clip-and-create` を 1 リクエストで完結。追加のページ遷移やダイアログ不要。 |
| **P2** | ✅ 適合  | スコープ付きトークンで、必要な権限だけを付与。リフレッシュ時も同一スコープを維持。                                 |
| **P3** | ✅ 適合  | 拡張 UI は「保存」「未保存」の状態表示程度に留め、重い操作は API に委譲。                                          |

## 4. 将来機能（即時／非同期）の検証

### 4.1 即時処理

| 機能                        | P1  | P2  | P3  | 備考                                                         |
| --------------------------- | --- | --- | --- | ------------------------------------------------------------ |
| clip-and-create（URL 保存） | ✅  | -   | ✅  | ワンクリックのコア                                           |
| 選択範囲保存                | ✅  | -   | ✅  | 1 クリックで選択範囲を送信                                   |
| 保存先ノート指定            | △   | -   | △   | ノート選択 UI が必要。デフォルトノート＋クイック選択で最小化 |

### 4.2 非同期ジョブ

| 機能           | P1  | P2  | P3  | 備考                                                   |
| -------------- | --- | --- | --- | ------------------------------------------------------ |
| AI 要約        | ✅  | -   | ✅  | 保存後バックグラウンド実行。ユーザー操作不要           |
| タグ提案       | ✅  | -   | ✅  | 同上                                                   |
| 重複検知       | ✅  | -   | ✅  | 同上                                                   |
| 関連ノート提示 | △   | -   | △   | 通知または次回開いた時の表示。ポップアップ肥大化に注意 |

### 4.3 リスクと軽減

| リスク                        | 軽減策                                                   |
| ----------------------------- | -------------------------------------------------------- |
| 即時 API が遅延すると UX 低下 | タイムアウト内に応答し、非同期ジョブは `job_id` で後追い |
| 保存先選択が複雑化            | デフォルトノートで即保存、高度な選択は任意               |
| オフライン時の動作            | 一時キューに保存し、オンライン復帰時に透過同期           |

---

## 5. 総合評価

| 観点         | 評価                                                            |
| ------------ | --------------------------------------------------------------- |
| **認証方式** | OAuth 2.0 + PKCE は 3 原則すべてに適合。                        |
| **API 設計** | capability スコープは最小権限と拡張性のバランスが取れている。   |
| **将来機能** | 即時／非同期の分類により、P1・P3 を維持したまま機能追加が可能。 |

**結論**: 現行プランはワンクリック保存・再認証透過・最小操作の UX 原則に適合している。実装時はタイムアウト設定・エラー時のフォールバック・オフラインキューの設計を仕様に明記すること。
