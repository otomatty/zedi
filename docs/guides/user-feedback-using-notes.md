# ノート機能を利用したユーザー意見・フィードバック収集の検討

開発中アプリでユーザーからの意見を収集するため、**アプリ本来のノート機能**を利用して「誰でも投稿できる場所」を実現できるかを調査した結果と、実現パターンをまとめます。

**権限設計の更新**: ノートの**閲覧権限**と**編集権限**は分離して設計する。詳細は [ノート権限設計（閲覧・編集の分離）](../specs/note-permissions-design.md) を参照。本ドキュメント内の「公開範囲」「誰が投稿できるか」は、その設計（visibility ＋ edit_permission）に従う。

---

## 1. 現在の実装状況・仕様

### 1.1 ノートの構造

| 概念 | 説明 |
|------|------|
| **ノート (notes)** | ページをまとめる「コンテナ」。1ノートに1人のオーナー（`owner_user_id`）が存在。 |
| **ノート内ページ (note_pages)** | 既存の「ページ」をノートに追加したもの。1つのページを複数ノートに追加可能。 |
| **ページ (pages)** | 実際のコンテンツ。各ページには所有者（`owner_id`）がいる。 |

- ノートに「投稿する」＝**ノートにページを追加する**行為に相当します。
- ノート内で「新規ページを追加」した場合、そのページの所有者は**ノートのオーナー**になります（仕様: `zedi-data-structure-spec.md` 方針B）。

### 1.2 閲覧権限（visibility）と編集権限（edit_permission）

ノート権限は **閲覧** と **編集** の2軸で設定する（[note-permissions-design.md](../specs/note-permissions-design.md)）。

- **閲覧権限（visibility）**: 誰がノートを見られるか。`private` / `restricted` / `unlisted` / `public`。
- **編集権限（edit_permission）**: 誰がノートにページを追加・編集できるか。`owner_only` / `members_editors` / `any_logged_in`。

「ログイン済みなら誰でも投稿可能」にするには、閲覧を public または unlisted にし、編集を **any_logged_in** に設定する。

**現状の API**: GET /api/notes/:id は「ログイン済みかつ（オーナーまたはメンバー）」の場合のみ成功する実装のため、public/unlisted でも非メンバーはノート取得できない。実装時に API を拡張し、visibility が public/unlisted のときは非メンバー・未ログインでも閲覧可能とする必要がある。

### 1.3 誰が「投稿」（ノートにページを追加）できるか

設計上（[note-permissions-design.md](../specs/note-permissions-design.md) に従う場合）:

- **オーナー**: 常に投稿・編集・削除が可能。
- **メンバー (editor)**: 常に投稿・編集が可能。
- **メンバー (viewer)**: 閲覧のみ。
- **非メンバー（ログイン済み）**: **edit_permission = any_logged_in** のとき、ページの**追加**が可能（既存ページの編集・削除はオーナー・メンバー editor のみ推奨）。

「ログイン済みなら誰でも投稿可能」にするには、ノートの編集権限を **any_logged_in** に設定する。現状の実装では edit_permission 未対応のため、API・DB の拡張が必要。

### 1.4 フロントの「ページを追加」の仕様

- **NoteView** の「ページを追加」は、**既に自分が持っているページ一覧から選んでノートに追加**する UI のみです。
- API には **「タイトルだけ指定してノート内に新規ページを作成する」** (`POST /api/notes/:id/pages` に `{ title }` を送る) が実装済みですが、**フロントでは未使用**です。
  - この API を使えば「ノート用の新規ページをその場で1本作成」するフローが可能です。

### 1.5 認証

- すべての `/api/*` は **Cognito JWT 必須**（未認証では 401）。
- 未ログインのまま「投稿だけ受け付ける」ような API は現状ありません。

## 2. 現状の仕様でできること・できないこと

| やりたいこと | 現状で可能か | 備考 |
|--------------|--------------|------|
| 開発者オーナーの「フィードバック用ノート」を1つ作る | ✅ 可能 | ノート作成後、visibility を public/unlisted/restricted のいずれかに設定可能。 |
| そのノートを「誰でも閲覧」する | ❌ 現状は不可 | 閲覧も API が認証＋オーナー/メンバー限定のため、未ログインや「メンバーでないログインユーザー」はノート内容を取得できない。 |
| **招待した人だけ**がログイン後にノートにページを追加（投稿）する | ✅ 可能 | 投稿してほしい人をメンバー（editor）で招待。招待された人は「自分のページをノートに追加」できる。 |
| **ログインしていれば誰でも**（メンバーでなくても）投稿できる | ❌ 不可 | 投稿権限はオーナー＋editor メンバーのみ。新機能追加が必要。 |
| **未ログイン**の人が投稿する | ❌ 不可 | 全 API が認証必須。別途「匿名フォーム」などの仕組みが必要。 |

---

## 3. アプリ本来の機能でコミュニケーションを提供するパターン

ノート機能を活かして「様々なユーザー同士がコミュニケーションをとれる環境」を用意するには、以下のような運用・拡張が考えられます。

### 3.1 現状の機能だけで運用する（招待制）

- **開発者（あなた）がオーナーのノートを1つ用意**（例: 「フィードバック・ご意見」）。
- そのノートの **メンバーに、意見を書いてほしい人を editor で招待**（メールアドレスで招待）。
- 招待されたユーザーは:
  1. アプリにサインアップ／ログインする。
  2. ノート URL（`/note/:noteId`）にアクセスする（ログインしていれば、自分がメンバーになっていれば閲覧・編集可能）。
  3. 「ページを追加」で**自分が既に持っているページ**をノートに追加する、という形で「投稿」できる。

**メリット**: 実装変更不要。  
**デメリット**: 「誰でも」ではなく「招待された人のみ」。招待作業が手間。未ログインでは見ることすらできない。

### 3.2 同じノートで「新規ページをノートに追加」UI を用意する（小改修）

- 上記と同じく、オーナーの「フィードバック用ノート」を用意し、メンバー（editor）を招待。
- フロントで **「このノートに新規ページを追加」** を実装する。既存 API `POST /api/notes/:id/pages` の `{ title }` を使い、ノート内にその場で新規ページを1本作成する。
- メンバーは「自分のページ一覧から選ぶ」ではなく、「タイトル（例: フィードバックの件名）を入力して投稿」というシンプルなフローにできる。

**メリット**: 既存のノート・ページ・API の仕様をそのまま利用。ユーザー体験が「投稿」に近くなる。  
**デメリット**: 投稿できるのはあくまで「オーナー＋メンバー（editor）」のみ。「誰でも」にはならない。

### 3.3 「ログイン済みなら誰でも投稿可能」に拡張する（採用方針・新機能）

**閲覧権限と編集権限の分離**（[note-permissions-design.md](../specs/note-permissions-design.md)）に従い、次のようにする。

- **閲覧権限（visibility）**: `public` または `unlisted` に設定したノートは、オーナー/メンバーでなくても閲覧可能（未ログインを許可するかは実装方針による）。
- **編集権限（edit_permission）**: **`any_logged_in`** に設定したノートでは、ログイン済みユーザーならメンバーでなくても「ページの追加」が可能。既存ページの編集・削除はオーナー・メンバー(editor) のみとすることを推奨。
- 必要な変更の例:
  - DB: `notes.edit_permission` カラムを追加（`owner_only` / `members_editors` / `any_logged_in`）。
  - API: 閲覧は `canAccessNote` で visibility が public/unlisted のとき非メンバーも許可。GET /api/notes/:id は認証オプション化し、未認証時は public/unlisted のみ返す。
  - API: ページ追加時に `edit_permission = any_logged_in` かつ閲覧可能なら、ログイン済みユーザーの追加を許可。
  - フロント: ノート作成・設定で「誰が投稿できますか？」として edit_permission を選択。公開ノートでは未ログインでも閲覧できるようにし、投稿ボタンはログイン時のみ表示。

**メリット**: ノート・ページをそのまま活用しつつ、「ログインすれば誰でも投稿できる」運用が可能。閲覧と編集を分けているため、「誰でも見られるが投稿は制限」も選べる。  
**デメリット**: API・DB・フロントの実装が必要。

### 3.4 未ログインでも「投稿」だけ受け付ける（別枠のフォーム）

- ノートとは別に、**フィードバック専用のフォーム**（例: ランディングやヘルプ内の「ご意見送信」）を用意。
- 送信時にバックエンドで「オーナー権限でノートにページを1本作成する」か、別テーブルに保存する。  
  ノートを使う場合は、作成したページを「フィードバック用ノート」に自動で追加するイメージ。

**メリット**: 未ログインでも意見を送れる。  
**デメリット**: ノートの「誰でも投稿」というより、**別機能**。スパム対策・モデレーションの検討が必要。

## 4. まとめと推奨

- **アプリ本来のノート機能だけで「誰でも投稿」**は、現状の仕様では**できません**。  
  - 投稿できるのは **オーナーとメンバー（editor）** のみで、かつ **すべて認証必須** です。
- **「様々なユーザー同士がコミュニケーションをとれる環境」**を、**今のノート機能の延長で**用意するなら:
  - **すぐ使うなら**: 上記 **3.1 または 3.2**（招待制＋必要なら「新規ページをノートに追加」UI）。
  - **ログイン済みユーザー全員に投稿してほしいなら**: **3.3**（public/unlisted ノートの「閲覧は誰でも／投稿はログイン済みなら誰でも」の拡張）。
- 未ログインの意見も集めたい場合は、ノートとは別の **3.4** のようなフォームを検討する形になります。

上記を踏まえ、「ノート＝コンテナ」「投稿＝ノートにページを追加」という現行のモデルを活かしつつ、運用（招待制）か、権限・公開仕様の拡張（3.3）、あるいはフォーム（3.4）のいずれを採用するか決めると整理しやすいです。
