# TDD 実践報告：AI 駆動開発におけるテストファーストの取り組み

**対象リポジトリ:** Zedi（Zero-Friction Knowledge Network）  
**報告期間:** 2025年10月〜2026年3月

---

## 1. はじめに：AI 駆動開発と品質という課題

AI 駆動開発（Cursor / Claude 等での補完・生成）では、**生成コードの品質**が課題になりやすいです。
「動くが意図とずれている」「エッジケースで壊れる」「リファクタで気づかず壊す」といった事象を抑えるため、**テストを先に書かせ、そのテストに合わせて実装を書く（テスト駆動開発・TDD）** やり方を採用しました。

今回のADCPにおける目的は次の二つです。

- **個人開発で TDD を実践し、TDD のスキルを磨くこと**
- **AI 駆動開発において何に気をつけるかの知見をまとめること**

本報告は、その取り組みの概要・成果・知見をまとめたものになります。

---

## 2. 取り組みの流れ

### 2.1 課題の整理

- AI に「この仕様で実装して」と依頼すると、**仕様の解釈違い**や**境界条件の漏れ**が発生しやすいです。
- 後からテストを書くと、**実装に引きずられたテスト**になりがちで、欠陥を検出しづらくなります。
- **リファクタや機能追加時に既存挙動が壊れていないか**を、人的チェックだけに頼るのは負荷が高くなります。

### 2.2 採用した方針：テストファースト

- **先にテストを書く** → そのテストを満たすように実装（AI）を行います。
- テストを **「仕様の契約」** として扱い、CI で毎回実行します。
- 単体テスト（Vitest）を中心に、E2E（Playwright）と Mutation testing（Stryker）で **テストの質** も可視化します。

これにより、

- AI に「このテストが通るように実装して」と指示しやすくなります。
- 実装後にテストが落ちれば **即フィードバック** が得られます。
- リファクタ時も **テストが安全網** になります。

という効果を狙いました。

## 3. やったこと

### 3.1 テストの規模

| 種別                 | 内容                                                                                                                                      |
| -------------------- | ----------------------------------------------------------------------------------------------------------------------------------------- |
| **単体・統合テスト** | Vitest。フロント（`src/`）、API（`server/api/`）、管理画面（`admin/`）をカバーしています。                                                |
| **テストファイル数** | 77 ファイル（`*.test.{ts,tsx}` 等）                                                                                                       |
| **テスト件数**       | **631 テスト**（`bun run test:run` で全件成功）                                                                                           |
| **E2E**              | Playwright。5 spec（検索・ページエディタ・WikiLink 作成ダイアログ・Web クリッパー・リンクページ）です。                                   |
| **カバレッジ**       | `bun run test:coverage` で v8 レポート（text / json / html）を取得し、CI で実行して artifact に保存しています。                           |
| **Mutation testing** | Stryker + Vitest。`src/lib/**`, `src/hooks/**` を対象とし、CI では PR 向けに限定（例: `dateUtils.ts`）、全量は Nightly で実行しています。 |

### 3.2 CI でのテストの位置づけ

- **CI（`.github/workflows/ci.yml`）**
  - Lint / Knip / Type Check に加え、**Unit Tests** ジョブで `bun run test:coverage` を実行しています。
  - カバレッジレポートを artifact として保存（7 日間）しています。
  - **Mutation (light)** ジョブで、限定対象の mutation テストを実行しています（`continue-on-error: true` のため merge はブロックしませんが、レポートで品質を確認できます）。
- **Nightly（`.github/workflows/nightly-mutation.yml`）**
  - 全対象の mutation テストを毎日実行し、レポートを 14 日間保持しています。

### 3.3 テストの設計方針

- **テストガイドライン**（`docs/guides/testing-guidelines.md`）
  - **テストは実装の前に書く**（TDD を徹底）し、**Mutation スコアを優先**、カバレッジは 80% 以上を目標としています。
  - describe / it は英語、仕様やエッジケースは**コメントで日本語併記**し、仕様書・ドキュメントへのパスをコメントで明示しています。
  - **モックの境界**として、認証・API・DB・ルーターはモックし、純粋ユーティリティはモックせず、単体は 1 ユニットに集中しています。
- **Mutation testing**
  - 閾値（high 80 / low 70 / break 65）を設定し、テストの「効き方」を数値で確認できるようにしています。

## 4. AI 駆動開発との相性の良さ（伝えたいポイント）

- **テストが仕様の契約になる**  
  「この describe / この it を満たす実装を書いて」と AI に依頼できます。曖昧な自然言語だけより意図が伝わりやすくなります。

- **即フィードバック**  
  実装後に `test:run` を流せば、仕様からのずれやエッジケースの漏れをすぐ検知できます。AI の出力を盲信せずに済みます。

- **リファクタの安全網**  
  既存テストが通る範囲で AI にリファクタを任せると、意図しない挙動変更を検出しやすくなります。

- **Living Documentation**  
  describe / it とコメントで「何を保証しているか」が残るため、後から自分や AI が仕様を読み直しやすくなります。

- **品質の可視化**  
  カバレッジと Mutation スコアで「テストが効いているか」を確認でき、AI にテストを書かせる際の品質チェックにも使えます。

## 5. 気をつけたこと・得た知見

- **モックの境界を決めておく**  
  認証・API・DB・ルーターはモックし、純粋ロジックはモックしない、とガイドラインで明文化しました。AI にテストを書かせる際も「この層はモックする」と指示しやすくなります。

- **テストは「振る舞い」で書く**  
  実装の細部（内部変数や 1 行の変更）に依存したテストは避け、**入力と期待される出力・副作用**で書くようにしました。そうすると AI が実装を差し替えてもテストが安定しやすくなります。

- **Mutation は段階導入**  
  いきなり全ファイルだと CI 時間が伸びるため、CI では限定対象（例: 1 ファイル）、全量は Nightly に分けました。まず 1 ファイルで時間を計測し、段階的に範囲を広げる方針を `docs/guides/testing-guidelines.md` に記載しています。

- **E2E は重要フローに絞る**  
  E2E は 5 spec に絞り、主要ユーザーフロー（検索・エディタ・WikiLink・Web クリッパー・リンクページ）をカバーしています。単体で担保できることは単体に任せ、E2E は「つなぎ」の確認に使う形にしました。

- **仕様・計画をドキュメント化する**  
  `docs/test-implementation-plan.md` や `docs/plans/20260216/page-editor-testing-proposal.md` のように、何をテストするか・どの順で広げるかを残しています。AI に「この Phase のテストを書いて」と依頼するときのコンテキストになります。

## 6. まとめ

- AI 駆動開発における**品質課題**を、「**テストを先に書かせ、そのテストに合わせて実装する**」TDD 的なやり方で緩和する取り組みを、個人開発のリポジトリで実践しました。
- 単体 631 テスト（77 ファイル）、E2E 5 spec、CI でのカバレッジ・Mutation（light）、Nightly での Mutation 全量という形で、**テスト基盤と実行結果がコードから確認できる**状態になっています。
- **AI 駆動開発との相性**としては、テストを仕様の契約として使うこと・即フィードバック・リファクタの安全網・Living Documentation・品質の可視化が、AI 駆動開発の品質向上に良い効果をもたらしていると感じています。

## 7. 今後の課題

- **未テスト領域の縮小**  
  `docs/test-implementation-plan.md` の Phase 2 以降（ミドルウェア・サービス・未テストルート・認証・ストレージ等）に沿って、テスト対象を段階的に広げていきます。特にセキュリティ・決済・データ整合性に直結する部分を優先します。

- **Mutation スコアの安定化と CI ゲート化**  
  現在、CI の Mutation (light) ジョブは `continue-on-error: true` のため merge をブロックしていません。スコアが安定してきた段階で、閾値（break 65 以上など）を満たさない場合は merge をブロックする運用に切り替えることを検討します。

- **E2E の拡充**  
  AI チャット・ノート（共有）・認証フロー・設定変更など、重要フローの E2E を追加し、主要ユーザー操作の端から端の保証を強めます。

- **テスト設計の見直し**  
  カバレッジ 80% を目標としつつ Mutation スコアを優先する方針に合わせ、「書くべきテスト」と「書かなくてよいテスト」の判断基準を、実例を踏まえてガイドラインに反映していきます。

## 8. さらに磨いていくスキル

- **TDD のリズム**  
  「Red → Green → Refactor」を短いサイクルで回す習慣を続け、テストを書くタイミングが遅れないようにします。AI に実装を依頼するときも、先にテストを渡してから依頼する流れを徹底します。

- **テスト設計（何をテストするか・何をモックするか）**  
  モックの境界や「振る舞い」で書くことの判断を、より一貫して行えるようにします。Mutation スコアが低い箇所を分析し、不足しているケースやアサーションの立て方を学びます。

- **Mutation スコアの読み方と改善**  
  レポートの survived mutant から「テストが効いていない変更」を把握し、テストの追加・修正に結びつけるスキルを高めます。閾値の意味（high / low / break）と、プロジェクトに合った目標値の設定のしかたを身につけます。

- **Living Documentation としてのテスト**  
  describe / it の名前とコメントで、仕様やエッジケースが読み手（自分・AI・他の開発者）に伝わるように書く力を磨きます。仕様書との対応をコメントで明示する習慣を維持します。
