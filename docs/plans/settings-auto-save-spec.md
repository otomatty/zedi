# 設定画面の自動保存 仕様

## 現在の実装状況

| 画面 | 保存対象 | 保存トリガー | 通知 |
|------|----------|--------------|------|
| **一般設定** (GeneralSettingsForm) | テーマ・フォント・言語（即時保存済み）、プロフィール（表示名・アバター） | 保存ボタン（saveGeneral + saveProfile） | useToast (Radix) |
| **AI設定** (AISettingsForm) | プロバイダー、モデル、APIキー、apiMode 等 | 保存ボタン | useToast |
| **ストレージ設定** (StorageSettingsForm) | 保存先トグル、プロバイダー、各プロバイダー用 config | 保存ボタン | useToast |

- 通知は現在すべて `useToast`（Radix）。Sonner は `App.tsx` で `<Toaster as Sonner />` として配置済み。
- 一般設定のテーマ・フォント・言語は前回対応で「変更即時保存」済み。未保存なのはプロフィールのみ。

## 仕様案（採用）

1. **自動保存のタイミング**
   - 入力のたびに保存すると API/ストレージ負荷とフラッシュが増えるため、**デバウンス 800ms** を採用する。
   - ユーザーが変更してから 800ms 無操作で保存を実行。続けて変更があればタイマーをリセット。

2. **対象**
   - **一般設定**: プロフィール（表示名・アバターURL）の変更時にデバウンス保存。テーマ・フォント・言語は従来どおり即時保存のまま。
   - **AI設定**: `updateSettings` が呼ばれるたびにデバウンス保存をスケジュール。
   - **ストレージ設定**: `updateSettings` / `updateConfig` が呼ばれるたびにデバウンス保存をスケジュール。

3. **通知**
   - 自動保存の成功・失敗は **Sonner** の `toast.success` / `toast.error` で表示する。
   - 既存の「保存しました」「保存に失敗しました」の文言を流用（i18n キーはそのまま利用）。

4. **保存ボタン**
   - 上記 3 画面から保存ボタンを削除する。テスト接続・リセット等のボタンは残す。

5. **returnTo の扱い**
   - AI設定・ストレージ設定では、保存成功時に `returnTo` があれば `navigate(returnTo)` している。自動保存成功後も同様に遷移する。
