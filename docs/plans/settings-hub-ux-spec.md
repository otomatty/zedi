# 設定ハブUX再設計 仕様書

**作成日:** 2026-03-10  
**対象:** 現在分割されている設定画面を `/settings` に集約し、何を設定できるかを直感的に理解できる構成へ改善する。

---

## 1. 背景

現状の設定UIは、`/settings` にカテゴリカードを並べ、各カテゴリを別ページに遷移して編集する構成になっている。

- `General`: `/settings/general`
- `AI`: `/settings/ai`
- `Storage`: `/settings/storage`

この構成には以下の課題がある。

1. **何を設定できるかが一覧だけでは分からない**
   - カテゴリ名と短い説明文だけでは、実際に変更できる項目や現在の設定状態が伝わりにくい。
2. **移動コストが高い**
   - 設定内容を確認するたびに別ページへ遷移する必要があり、比較や見直しがしづらい。
3. **設定の粒度が揃っていない**
   - `General` は複数カードの集合、`AI` と `Storage` は条件分岐の多いフォームであり、すべてを同じ「詳細ページ」パターンで扱うと理解しづらい。
4. **発見性が低い**
   - 「AI はサーバーモードか自前APIキーか」「Storage はどこに保存されるか」など、ユーザーが最初に知りたい情報が設定画面トップで見えない。

---

## 2. 目的

- 設定画面を開いた時点で、**何を設定できるか** と **現在どう設定されているか** が把握できること
- 設定確認と編集のための画面遷移を減らし、**比較・見直し・調整をしやすくする**こと
- 情報量の多い `AI` / `Storage` 設定でも、初見ユーザーが迷わない構成にすること
- 単なる機能追加ではなく、設定体験全体をより洗練されたものにすること

---

## 3. スコープ

### 対象

- `/settings` の情報設計と画面構成の再設計
- `General` / `AI` / `Storage` の表示統合
- 設定サマリー表示
- セクション移動導線
- 一部セクションの開閉UI
- URLでのセクション状態の表現
- 旧設定画面・旧ルート・不要になった関連ファイルの削除

### 非対象

- 各設定項目の保存ロジックそのものの大幅変更
- 新しい設定項目の追加
- Note単位の設定（`/note/:noteId/settings`）の再設計
- アカウント課金や購読設定の統合

---

## 4. 採用方針

### 4.1 採用するUI方針

**`/settings` を単一ページの設定ハブに再設計し、カテゴリごとに「要約付きセクション」を並べる。**

今回の改善は**段階移行ではなく、一括置き換え**で行う。

- 旧UIとの後方互換性は考慮しない
- 旧ルートを残すための暫定実装は行わない
- 新UIに不要なページ、ルート、ラッパー、文言、分岐は削除する
- 「今後使うかもしれない」理由で未使用コードを残さない

各カテゴリは以下の2層で構成する。

1. **サマリー層**
   - そのカテゴリで何を設定できるか
   - 現在の主要状態
   - そのセクションへ移動する導線
2. **編集層**
   - 実際の設定フォーム
   - 必要に応じて高度な項目だけ開閉

### 4.2 アコーディオンの位置づけ

アコーディオンは採用可能だが、**トップレベル全体をアコーディオンだけで構成しない**。

理由:

- 3カテゴリしかないため、最上位で完全に折りたたむと全体像が逆に見えなくなる
- `AI` と `Storage` は条件分岐が多く、初回利用時に「何があるのか」を見せる方が重要
- 高度な設定・補助説明・危険操作の折りたたみに使う方が適している

したがって、採用するのは **「1ページ集約 + セクション表示 + 部分アコーディオン」** である。

---

## 5. 非採用案

### 5.1 全カテゴリを単純なアコーディオンにする案

**評価:** 次点だが不採用

メリット:

- 1ページに集約できる
- 実装しやすい
- モバイルで縦に並べやすい

デメリット:

- 最初に見える情報量が少なく、何を設定できるかが依然として分かりにくい
- 開くまで中身が見えず、設定の発見性が十分に改善しない
- `AI` / `Storage` の複雑さをそのまま隠してしまい、初見理解を助けにくい

### 5.2 現状の別ページ構成を維持し、トップカードだけ改善する案

**評価:** 効果が限定的なため不採用

メリット:

- 既存実装への影響が小さい
- 開発コストが低い

デメリット:

- 遷移コストが残る
- 複数カテゴリの比較がしにくい
- 設定全体の見取り図ができない

### 5.3 タブUIにする案

**評価:** 不採用

メリット:

- セクション切り替えが明確
- デスクトップでは見栄えが良い

デメリット:

- モバイルで扱いづらい
- 非表示タブの情報が見えず、何を設定できるかが分かりにくい
- URL・履歴・初見理解の面で設定ハブとして弱い

---

## 6. 画面構成

## 6.1 ページ全体

`/settings` は以下の構成とする。

1. ヘッダー
   - 戻る導線
   - ページタイトル `設定`
2. 設定ハブ概要
   - 短い説明文
   - 主要カテゴリのサマリーカードまたはジャンプリスト
3. セクション本体
   - `General`
   - `AI`
   - `Storage`
4. 補足領域
   - 高度な設定
   - 危険な操作
   - 関連ガイドへのリンク

### 6.2 サマリーカード / ジャンプリスト

ページ上部にカテゴリごとの要約を表示する。

表示内容の例:

| セクション | 表示する要約例                             |
| ---------- | ------------------------------------------ |
| General    | テーマ、フォントサイズ、言語、プロフィール |
| AI         | 利用モード、選択中モデル、接続状況         |
| Storage    | 保存先、選択中プロバイダー、接続状況       |

サマリーには以下を含める。

- セクション名
- 何を設定できるかの短い説明
- 現在の主な設定状態
- クリック時に該当セクションへスクロールする導線

### 6.3 セクション本文

各セクションは以下の形式で統一する。

- セクションタイトル
- セクション説明
- 現在の状態を一行で表示するサブサマリー
- 実際の設定フォーム
- 必要に応じて補助説明、テスト、リセット、高度な設定

---

## 7. セクション別仕様

### 7.1 General セクション

表示対象:

- プロフィール
- 表示設定
- ツアー再実行
- 言語
- About
- データ管理（サインイン時のみ）

サマリー例:

- テーマ: ダーク
- フォントサイズ: 16px
- 言語: 日本語
- プロフィール: 表示名あり

UI方針:

- `GeneralSettingsForm` の既存カード構成は活かす
- ただし単独ページではなく、`/settings` 内の最初の主要セクションとして表示する
- 初見理解を優先し、ここはトップレベルで折りたたまない

### 7.2 AI セクション

表示対象:

- サーバーモード / 自前APIキーの切り替え
- モデル選択
- APIプロバイダー選択
- APIキー入力
- 接続テスト
- リセット

サマリー例:

- モード: サーバー利用
- モデル: Claude Sonnet
- 接続状態: 設定済み

UI方針:

- 最初に「現在どのモードか」を大きく見せる
- `useOwnKey` が `false` の場合は「Zedi側の設定で利用中」を簡潔に表示
- `useOwnKey` が `true` の場合のみ、プロバイダー・APIキー・モデル選択を表示
- APIキーの取得先リンクや補助説明は「詳細を見る」で折りたたみ可能にする
- リセットはメイン設定から距離を置き、セクション下部に配置する

### 7.3 Storage セクション

表示対象:

- デフォルト保存先 / 外部ストレージの切り替え
- プロバイダー選択
- プロバイダー別認証情報
- 接続テスト
- セットアップガイド
- リセット

サマリー例:

- 保存先: デフォルト
- 外部ストレージ: GitHub
- 接続状態: 未テスト / 成功 / 失敗

UI方針:

- まず「どこに保存されるか」を一文で明確に出す
- `preferDefaultStorage` が有効な場合は、外部ストレージ設定を初期表示で隠す
- 外部ストレージ利用時のみ、プロバイダー選択と詳細フォームを表示する
- プロバイダーごとの詳細入力欄は必要最小限だけ初期表示し、長い補助説明やURL説明は折りたたみ可能にする
- セットアップガイドへのリンクは常に視認可能にする

## 8. ルーティング方針

### 8.1 採用案

設定編集の主導線は `/settings` に集約する。

| パス        | 役割               |
| ----------- | ------------------ |
| `/settings` | 設定ハブ兼編集画面 |

### 8.2 旧ルートの扱い

既存の以下ルートは**廃止**する。

- `/settings/general`
- `/settings/ai`
- `/settings/storage`

方針:

1. 旧ルートは残さない
2. リダイレクト前提の暫定対応は行わない
3. `App.tsx` から対象ルートを削除する
4. 旧ルート専用のページコンポーネントも削除する

---

## 9. URL状態

設定画面は、どのセクションを見ているかをURLに反映できるようにする。

採用例:

- `/settings?section=general`
- `/settings?section=ai`
- `/settings?section=storage`

必要に応じて以下も許可する。

- `/settings?section=storage&provider=github`

要件:

- サマリーカードからの移動で `section` を更新する
- 直接URLアクセス時は該当セクションを初期表示位置として扱う
- 将来アコーディオン開閉状態を持たせる場合も、必要最小限だけURL反映する

---

## 10. レイアウト・操作要件

### 10.1 デスクトップ

- 上部サマリーから目的セクションへすぐ移動できること
- セクション間の境界が視覚的に明確であること
- 長いフォームでも読み始め位置を見失いにくいこと

### 10.2 モバイル

- 縦積み前提で破綻しないこと
- タブ前提にしないこと
- 長い説明文は必要に応じて折りたたみ可能であること
- セクション移動がしやすいよう、上部ジャンプリストを配置すること

### 10.3 視認性

- 各セクションに「何ができるか」を短文で明示すること
- 現在値のサマリーは専門用語だけでなく意味が伝わる文面にすること
- 危険操作は通常の設定群と視覚的に分離すること

---

## 11. コピー方針

設定画面の文言は、説明のための説明にならないよう、以下の原則に沿う。

1. **カテゴリ名だけで終わらせない**
   - 例: `AI設定` だけでなく `AIモデル、接続方法、APIキーを設定` を添える
2. **現在状態を短く言い切る**
   - 例: `サーバー利用中` `GitHubに保存` `日本語を表示中`
3. **補助説明は必要箇所だけ**
   - 初期表示ですべてを長文説明しない
4. **危険操作は意図が分かる文言にする**
   - `リセット` だけでなく、何が初期化されるか分かるようにする

---

## 12. コンポーネント構成案

### 12.1 新規または再構成候補

| 種別           | ファイル                                       | 役割                                          |
| -------------- | ---------------------------------------------- | --------------------------------------------- |
| ページ         | `src/pages/Settings.tsx`                       | 設定ハブ本体。サマリー + セクション本文を表示 |
| コンポーネント | `src/components/settings/SettingsOverview.tsx` | 上部ジャンプリスト / サマリー表示             |

| コンポーネント | `src/components/settings/CollapsibleHelp.tsx`  | 補助説明や高度な設定の折りたたみ              |

### 12.2 再利用する既存コンポーネント

- `GeneralSettingsForm`
- `AISettingsForm`
- `StorageSettingsForm`

方針:

- 既存フォームのロジックは極力再利用してよい
- ただし、旧ページ前提の余白・見出し・戻る導線など不要な構造は統合時に整理する
- 新構成で不要になったラッパーや重複UIはその場で削除する

### 12.3 削除対象

新UIへの置き換え時に、以下は削除対象とする。

| 種別   | ファイル / 対象                                                       | 削除方針     |
| ------ | --------------------------------------------------------------------- | ------------ |
| ページ | `src/pages/GeneralSettings.tsx`                                       | 廃止         |
| ページ | `src/pages/AISettings.tsx`                                            | 廃止         |
| ページ | `src/pages/StorageSettings.tsx`                                       | 廃止         |
| ルート | `App.tsx` 内の `/settings/general` `/settings/ai` `/settings/storage` | 削除         |
| UI     | 旧 `Settings.tsx` のリンクカード一覧構成                              | 全面置き換え |

補足:

- 削除対象に紐づく import は必ず整理する
- 未使用の翻訳キー、補助文言、スタイルが発生した場合は合わせて削除する
- 旧画面専用のテストが存在する場合は、新UIに合わせて更新または削除する

## 13. 状態管理・保存方針

この再設計は主にUI構造の変更であり、基本的には既存保存ロジックを維持する。

- `General`: 既存 `useGeneralSettings` / `useProfile`
- `AI`: 既存 `useAISettings`
- `Storage`: 既存 `useStorageSettings`

注意点:

- 1ページ集約後も、各セクションの保存責務は分離する
- `AI` / `Storage` の自動保存や `returnTo` の挙動は、旧詳細ページ遷移前提のまま残さず、新しい `/settings` 中心の導線へ合わせて整理する
- サマリー表示は各フックの状態から導出し、別の永続化データは持たない

---

## 14. アクセシビリティ要件

- セクションジャンプは可能なら `button` より `a` / `Link` 相当で表現し、URLと整合させる
- アイコンのみの操作には `aria-label` を付与する
- 折りたたみUIには展開状態を明示する
- 長いページでも、見出し構造が崩れないよう `h1` -> `h2` -> `h3` を守る
- キーボードだけで各セクションへ移動できること

---

## 15. 実装方針

今回の対応は、既存設定画面を段階的に移行するのではなく、**新UIへ一気に置き換える**。

実装順序:

1. `/settings` を新しい設定ハブへ作り替える
2. `General` `AI` `Storage` を1ページ内へ統合する
3. 上部サマリー、セクション導線、必要な開閉UIを実装する
4. 旧設定ページと旧ルートを削除する
5. 不要になった import、未使用コンポーネント、翻訳キー、テストを整理する

禁止事項:

- 互換維持のためだけのリダイレクト追加
- 一時しのぎの旧UI残置
- 実際には参照されない未使用ファイルの放置

---

## 16. 受け入れ基準

- ユーザーが `/settings` を開いた時点で、`General` `AI` `Storage` の3カテゴリで何を設定できるか把握できる
- 各カテゴリについて、現在の主要設定状態がサマリーとして見える
- 設定確認のためにカテゴリ別詳細ページへ遷移しなくても、主要な設定内容を同一画面で閲覧・編集できる
- `AI` と `Storage` の複雑な設定でも、初見ユーザーが「まず何を決めるか」を理解できる
- モバイルでも縦スクロール中心で破綻なく操作できる
- 補助説明や高度な設定は必要なときだけ展開できる
- URLまたは導線上で、対象セクションへ直接移動できる
- `/settings/general` `/settings/ai` `/settings/storage` に依存するコードが残っていない
- 旧設定画面専用の不要ファイル・未使用import・未使用翻訳キーが整理されている

---

## 17. 実装メモ

- 現状 `Settings.tsx` はリンクカードの一覧であり、ここを全面的に役割変更する
- `AISettingsForm` / `StorageSettingsForm` は単一 `Card` 前提のため、設定ハブに埋め込んだ際の余白・見出し重複に注意する
- `GeneralSettingsForm` は複数カード構成のため、設定ハブのセクション構造と相性が良い
- 上部サマリーには「現在値が未設定の場合の表示ルール」を先に決める必要がある
- 旧ページ削除に伴い、ページ単位のヘッダーや戻る導線が二重に残らないように調整する
- 実装完了時には、設定UI関連ファイルを再走査し、不要ファイルが残っていないことを確認する

未設定時の表示例:

- AI: `未設定` / `デフォルト利用`
- Storage: `デフォルト保存` / `外部ストレージ未接続`
- Profile: `未入力`

---

## 18. まとめ

今回の設定UI改善では、**設定をページで分ける発想から、設定全体を俯瞰できるハブへ変える**ことを重視する。

採用するのは、単純な全面アコーディオンではなく、以下の構成である。

- `/settings` に設定を集約
- 上部にカテゴリ別サマリーを表示
- 下部に `General` `AI` `Storage` の主要セクションを並べる
- アコーディオンは補助説明や高度な設定に限定して使う
- 後方互換性は持たず、旧設定ページ・旧ルート・不要コードは削除する

この方針により、「どこで何を設定できるのか分からない」という現在の課題を解消しつつ、今後設定項目が増えても破綻しにくい設計にできる。
