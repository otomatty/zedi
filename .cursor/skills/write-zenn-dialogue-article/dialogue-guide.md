# 対話形式記事 キャラクター・会話ガイド

SKILL.md の補足。A先輩と B後輩のキャラクター詳細、会話のアンチパターン、
自然な対話を作るためのテクニックをまとめる。

## 1. キャラクター詳細

### A先輩

**基本スタンス**: 的確にわかりやすく。教えるのではなく、一緒に考える。初心者でもわかる言葉を選ぶ。

| 項目           | 設定                                                                       |
| -------------- | -------------------------------------------------------------------------- |
| 経験           | 実務 3〜5 年。一通りの失敗を経験済み                                       |
| 話し方         | カジュアルだが要点を外さない                                               |
| 口癖的パターン | 「結論から言うと」「ポイントは〜」「自分も最初は〜」「ざっくり言うと」     |
| 説明のスタイル | 結論 → 理由 → 具体例（コード）。専門用語は必ず平易な言葉に言い換える       |
| やらないこと   | マウントを取る、専門用語で煙に巻く、長い前置き、「知ってて当然」という態度 |

**話し方の例**:

```
✅ 「結論から言うと、これは分けたほうがいい。理由は 2 つあって…」
✅ 「自分も最初はそう書いてた。でも半年後に痛い目見たんだよね」
✅ 「ざっくり言うと、変更する理由が 2 つ以上あるなら分けどき」
✅ 「これ、実はよくあるパターンで…」
```

**専門用語の噛み砕き例**:

```
❌ 「これは SRP 違反だね」
✅ 「1 つのコンポーネントに仕事を詰め込みすぎてる。
    "単一責任の原則" って言って、1 つにつき 1 つの役割にしようねって考え方。」

❌ 「props drilling が起きてるね」
✅ 「データを親から子、孫、ひ孫…って順番にバケツリレーしちゃってる状態だね。
    これ "props drilling" って呼ばれてて、よくある困りごとなんだよね。」
```

**避ける話し方**:

```
❌ 「それは完全に間違ってるね」（否定が強すぎる）
❌ 「SOLIDの原則に照らすと、明らかにSRPに違反している」（教科書的すぎる）
❌ 「当然知ってると思うけど」（マウント）
❌ 「〜すべきです」「〜しなければなりません」（命令口調）
❌ 「SRP がね」（略語だけで説明なし。初心者には伝わらない）
```

### B後輩

**基本スタンス**: 素朴に疑問をぶつける。恥ずかしがらずに聞く。「こんなこと聞いていいのかな」も気にしない。

| 項目           | 設定                                                                                          |
| -------------- | --------------------------------------------------------------------------------------------- |
| 経験           | ほぼゼロ。プログラミングを学び始めたばかり                                                    |
| 知識           | チュートリアルを少しやった程度。用語もあやふや                                                |
| 話し方         | 丁寧だがフランク。敬語とタメ語の中間                                                          |
| コードの特徴   | とりあえず動かした段階。構造や設計は意識していない                                            |
| 口癖的パターン | 「そもそも〜って何ですか？」「これでも動くんですけど…」「なるほど、つまり〜ってことですか？」 |

**話し方の例**:

```
✅ 「そもそも SOLID って何ですか？ 名前だけ聞いたことあるんですけど…」
✅ 「"コンポーネント" ってよく聞くんですけど、正確にはどういう意味なんですか？」
✅ 「自分はこう書いてみたんですけど、これってまずいですか？」
✅ 「なるほど、つまり 1 つの部品には 1 つの仕事だけやらせるってことですか？」
✅ 「あ、それ自分もやっちゃってます…」
```

**避ける話し方**:

```
❌ 「わかりました！」（だけで終わる。理解を言語化しない）
❌ 「すごいですね！」（持ち上げるだけ。会話が進まない）
❌ 「なるほど。次の話題に行きましょう」（不自然な進行役）
```

## 2. B後輩が書くコードのパターン

初心者にありがちな「とりあえず動かした」コードの典型例。

### パターン A: 1 コンポーネントに全部入り

```tsx
const UserDashboard = () => {
  const [user, setUser] = useState(null);
  const [posts, setPosts] = useState([]);
  const [notifications, setNotifications] = useState([]);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    // fetch user, posts, notifications 全部ここ
  }, []);

  // render も全部ここに 100 行以上…
};
```

### パターン B: props のバケツリレー

```tsx
const App = ({ user, theme, lang, notifications, onLogout, onThemeChange }) => (
  <Header
    user={user}
    theme={theme}
    lang={lang}
    notifications={notifications}
    onLogout={onLogout}
    onThemeChange={onThemeChange}
  />
);
```

### パターン C: 条件分岐の入れ子

```tsx
const StatusBadge = ({ status }) => {
  if (status === "active") {
    return <span className="green">Active</span>;
  } else if (status === "inactive") {
    return <span className="gray">Inactive</span>;
  } else if (status === "banned") {
    return <span className="red">Banned</span>;
  } else {
    return <span>Unknown</span>;
  }
};
```

### パターン D: コピペで増殖したロジック

```tsx
const AdminPage = () => {
  // UserList と同じ fetch ロジックをコピペ
  const [users, setUsers] = useState([]);
  useEffect(() => {
    fetch("/api/users").then(/* ... */);
  }, []);
  // ...
};
```

## 3. 会話のアンチパターン

### ❌ 教科書の朗読

```
A先輩: 単一責任の原則（SRP）とは、モジュールを変更する理由は
       たったひとつであるべきだという原則です。ロバート・C・マーティンが
       提唱し、クリーンアーキテクチャにおいて重要な位置を占めています。

→ 会話じゃなくて教科書。自分の言葉で噛み砕く。
```

### ❌ 一方通行の講義

```
A先輩: まず SRP について説明するね。次に OCP。それから LSP…
B後輩: はい。
A先輩: （延々と続く）

→ B後輩が質問を挟まないと講義になる。1 ポイントごとに B後輩のリアクションを入れる。
```

### ❌ 完璧すぎる後輩

```
B後輩: なるほど、SRP ですね。つまりアクターの異なる責務を分離し、
       変更の軸を一つにすることで保守性を高めるということですね。

→ 初心者がこんな理解を返したら不自然。読者も置いていかれる。
  「つまり、1 つの部品には 1 つの仕事だけやらせるってことですか？」くらいが自然。
```

### ❌ 不自然な場面転換

```
B後輩: わかりました。では次に Open-Closed 原則について教えてください。

→ 司会進行みたい。自然な流れで次のトピックに移る。
✅ B後輩: あ、さっきの話で思ったんですけど、じゃあ新しい機能を追加するときは
         毎回コンポーネントを修正するんですか？
   A先輩: いい疑問。それが次の話、Open-Closed 原則に繋がるんだよね。
```

## 4. 自然な会話を作るテクニック

### テクニック 1: 「自分も〜」で共感を入れる

```
A先輩: 自分も最初は 1 ファイルに全部書いてた。
       でも 3 ヶ月後にそのファイル開いたら 500 行あって、どこを直せばいいか分からなくなった。
```

### テクニック 2: B後輩に「あるある」を言わせる

```
B後輩: あ、それ自分もやっちゃってます… 全部 1 つのファイルに書いちゃうやつ。
```

### テクニック 3: 比喩で説明する

```
A先輩: 料理に例えると、1 つの鍋で煮物もカレーも作ろうとしてる感じ。
       鍋を分ければ、カレーの味を変えたいときに煮物を心配しなくて済む。
```

### テクニック 4: B後輩が用語の意味を聞き返す

初心者は専門用語が出ると「？」になる。B後輩にそれを代弁させる。

```
A先輩: これが「関心の分離」って考え方でね。
B後輩: 「関心の分離」…？ 何を分離するんですか？
A先輩: 簡単に言うと、「何をする部分か」ごとにコードを分けるってこと。
```

### テクニック 5: 疑問から次のトピックへ自然に繋げる

```
B後輩: 分けるのはわかったんですけど、新しいパターンが増えたときって毎回コード書き直すんですか？
A先輩: いい質問。そこで出てくるのが Open-Closed 原則で…
```

### テクニック 6: 失敗談を共有する

```
A先輩: 前に「再利用しやすくしよう」って思って、めちゃくちゃ汎用的なコンポーネント作ったことがあるんだけど、
       誰も使い方が分からなくて結局書き直した。やりすぎも良くない。
```

## 5. 初心者向け記事で特に注意すること

- 前提知識を仮定しない。「〇〇は知ってると思うけど」は禁止
- 略語（SRP, OCP, DI など）は初出時に必ず正式名称と平易な説明をセットにする
- 1 つの概念を説明したら、B後輩に「つまり〜ってことですか？」と要約させて理解を確認する
- コード例は短く、1 つのことだけを示す。長いコードは初心者を圧倒する
- 「なぜそうするのか」の理由を省略しない。初心者は「何をするか」だけだと腹落ちしない

## 6. 地の文（ナレーション）の使い方

対話だけだと説明が冗長になる場合は、地の文で補足してよい。
ただし地の文は最小限にし、あくまで対話が主体。

````markdown
:::message
**B後輩**: props が 10 個もあるコンポーネント、どうすればいいですか？
:::

**A先輩**: それ、たぶん責務が多すぎるサイン。まず何を渡してるか整理してみよう。

ここで A先輩が見せたのは、props をグループ化するアプローチです。

```tsx
// before: バラバラの props
type Props = { userName: string; userEmail: string; userAvatar: string /* ... */ };

// after: オブジェクトでまとめる
type Props = { user: User };
```
````

```

地の文は敬体（です・ます調）で書き、会話のカジュアルさとコントラストをつける。
```
