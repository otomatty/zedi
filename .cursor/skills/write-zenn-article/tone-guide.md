# Zenn 記事トーンガイド — 詳細リファレンス

SKILL.md の補足。AI が生成しがちな文章パターンと、人間らしい書き換え例を網羅する。

## 1. AI 臭のある文章パターンと書き換え

### パターン A: 大げさな前置き

```
❌ 近年、AI駆動開発が急速に普及し、多くの開発者がAIツールを活用して
   生産性を大幅に向上させています。本記事では、そのような環境における
   テスト品質の課題と、私たちが取り組んだ解決策について包括的に解説します。

✅ Cursor を使い始めて半年くらい経ちます。
   コードを書くスピードは上がりました。でもテストの質は上がっていませんでした。
```

ポイント: 主語を「自分」にして、いきなり体験から始める。

### パターン B: 箇条書きの羅列で終わる

```
❌ 以下のメリットがあります：
   - コード品質の向上
   - 開発効率の改善
   - バグの早期発見
   - チーム間のコミュニケーション促進

✅ 一番良かったのは、AI への指示が具体的になったことです。
   「テストを強化して」じゃなくて「この Survived ミュータントを殺して」と
   言えるようになりました。指示の解像度が上がると、AI の出力も変わります。
```

ポイント: 抽象的なメリット列挙ではなく、1 つを深掘りして体験で語る。

### パターン C: 受動態の多用

```
❌ テストの品質が担保されます。仕様の整合性が確保されます。
   開発プロセスの改善が図られます。

✅ テストの質が数字で見えるようになりました。
   仕様がコードの中にあるので、ズレようがありません。
```

ポイント: 「〜される」「〜が図られる」を「〜しました」「〜になりました」に。

### パターン D: 形式的なまとめ

```
❌ 以上のことから、TDDとMutation Testingの組み合わせは、
   AI駆動開発における品質保証において極めて有効であると結論づけることができます。
   今後もさらなる改善を続けていく所存です。

✅ やってみて思ったのは、AI が強力になるほど「テストの質を測る仕組み」が要るということです。
   カバレッジだけ見ていたら気づけなかった穴が、Mutation Test で見えるようになりました。
   まだサーバーサイドには入れていないので、次はそこに手をつけます。
```

ポイント: 「結論づけることができます」は書かない。「やってみて思ったのは」で始める。

### パターン E: 定義文から始まる

```
❌ Mutation Testingとは、ソースコードに意図的な変更を加え、
   テストスイートがその変更を検出できるかどうかを評価する
   ソフトウェアテスト技法です。

✅ Mutation Testing は、コードをわざと壊してみて、
   テストがそれに気づくかどうかを調べる手法です。
   「テストのテスト」と言うと分かりやすいかもしれません。
```

ポイント: 辞書的な定義を避け、自分の言葉で噛み砕く。比喩を添えると良い。

## 2. 接続のリズム

### 段落の切り替えパターン

同じ接続詞を 2 回以上連続で使わない。以下を循環させる。

```
（事実を述べる）
→ でも / ただ（逆接）
→ そこで（行動）
→ 結果として / やってみると（結果）
→ ちなみに / 補足すると（余談）
→ 次にやったのは（展開）
```

### 段落の長さ

- 1 段落は 3〜5 文
- 6 文以上になったら段落を分ける
- 1 文の段落も効果的に使える（強調したいとき）

```
✅ Stryker を走らせました。

   結果は衝撃的でした。109 行のテストでは、ほとんどのミュータントが生き残っていました。
```

## 3. タイトルの付け方

### 良いタイトル

具体的で、読者が内容を想像できるもの。

- ✅ 「AI駆動開発で"通るだけのテスト"を撲滅する — TDD × Mutation Test 実践記」
- ✅ 「カバレッジ100%なのにバグを見逃すテスト、Strykerで全部見つけた」
- ✅ 「テスト109行で安心していた自分に、Mutation Testが現実を教えてくれた」

### 悪いタイトル

抽象的で、何の話か分からないもの。

- ❌ 「AI駆動開発におけるテスト品質向上の取り組みについて」
- ❌ 「TDDとMutation Testingの導入」
- ❌ 「テスト戦略の改善」

### タイトル構造のパターン

| パターン    | 例                                    |
| ----------- | ------------------------------------- |
| 事実 + 手段 | 「〜だった → 〜で解決した」           |
| 意外性      | 「カバレッジ100%なのに〜」            |
| 数字 + 変化 | 「テスト109行が887行になった話」      |
| 問い        | 「AI が書いたテスト、信用していい？」 |

## 4. 引用の使い方

外部記事を引用するとき。

```
❌ ryuka氏は以下のように述べています。

✅ ryuka氏の記事にこんな一節がありました。
```

引用ブロックの後には、自分の感想を 1〜2 文で添える。

```
✅ > カバレッジ100%でもミューテーションスコアが38%だった
>
> — ryuka氏の記事より

これを読んで、自分のプロジェクトはどうなんだろう、と気になりました。
```

## 5. 技術的な正確性のバランス

- 正確さは大事だが、読みやすさとのトレードオフがある
- 細かい注釈は本文ではなく脚注的に書く
- 厳密な定義が必要な場合は「ざっくり言うと」で平易な説明を先に出し、正確な説明を括弧や補足で添える

```
✅ ざっくり言うと、コードをわざと壊してテストが気づくか調べる手法です。
   （正確には、ソースコードにミュータント＝小さな変更を注入し、
   テストスイートの検出能力を定量的に測定します）
```
