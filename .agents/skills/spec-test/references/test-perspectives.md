# テスト観点リファレンス / Test Perspectives Reference

`spec-test` / `test-inventory` 全エージェントの共通判断基準。

## 0. 大原則 — 振る舞いを検証する

**実装の詳細ではなく、外から観測できる振る舞いを検証する。**

- 判定基準: 「実装を別の書き方に書き直してもテストが通るか？」通るなら良いテスト。
- UI はアクセシブルなクエリ（Testing Library なら `getByRole` / `getByText` 等）。クラス名・DOM 構造に依存しない。
- 内部関数の呼び出し回数や順序は、それ自体が仕様（例: 課金 API は 1 回だけ呼ぶ）でない限り検証しない。

## 1. ケース設計の 4 技法（この順に適用する）

### ① 同値分割

入力を「同じ振る舞いをするグループ」に分け、各グループから代表値を 1 つ選ぶ。
最低限のグループ: 正常系 / バリデーションエラー / 認可エラー / 外部依存の失敗。

### ② 境界値分析

グループが切り替わる位置の**両側**を必ず取る。バグ（off-by-one、`<` と `<=` の取り違え）は境界に集中する。

- 数値の上下限: 上限ちょうど / 上限 + 1 / 下限ちょうど / 下限 − 1
- 長さ: 空 / 1 件 / 上限ちょうど / 上限超え
- 閾値比較: 閾値ちょうどの値と 1 つ外れた値
- 境界が定数の場合、テストは**定数を import して使う**（境界変更時に追従）

### ③ デシジョンテーブル

条件が 2 つ以上絡む場合（権限 × 公開状態、プラン × 機能フラグなど）は組み合わせ表を作り、
テーブル駆動テスト（Vitest `it.each`、pytest `@pytest.mark.parametrize` 等）で実装する。

### ④ 状態遷移

状態を持つ対象は「遷移できる組み合わせ」と「**遷移できないはずの組み合わせ**」の両方を検証する。

## 2. 見落としやすい観点チェックリスト

- [ ] **空・ゼロ・null**: 空配列、空文字、0 件ヒット、undefined / None
- [ ] **認可**（認証とは別）: 他人のリソースへのアクセスが拒否されるか。未認証だけで満足しない
- [ ] **エラー後の状態**: 外部依存が失敗したとき、フォールバック・通知・ロールバックが仕様通りか
- [ ] **冪等性・重複実行**: 同じ操作を 2 回実行したら？（保存連打、Webhook 再送）
- [ ] **時間**: タイムゾーン、月末・年末。fake timers / 固定日時で再現
- [ ] **文字**: マルチバイト、絵文字、サロゲートペア（文字数・切り詰め・正規化で頻出）
- [ ] **並行・競合**: 共同編集、キャッシュ無効化、デバウンス/スロットル

## 3. テストの構造

- **AAA（Arrange-Act-Assert）** を空行で区切る。準備が長い場合はビルダー/ヘルパーへ
  （**プロジェクト既存の** mock ヘルパーがあればそれに合わせる。無ければ Arrange を明示的に書く）。
- **テスト名は仕様の文章**: `it("上限 200 を超える limit は 200 に丸める")`。
- **1 テスト 1 関心事**。アサーションは複数でよいが、検証する振る舞いは 1 つ。
- **テスト内のロジック禁止**: if / for / 期待値の計算を書かない。期待値はハードコードする。
- describe / context はコンポーネント・関数単位。リポジトリが日英併記コメントを求める場合のみ従う。

## 4. 信頼性 — 嘘をつかないテスト

- **決定的にする**: 現在時刻 / 乱数は固定、ネットワークはモック。
  固定 sleep より条件待ち（`waitFor`, `expect(...).toBeVisible()` 等）。
- **独立にする**: テスト間で状態を共有しない。実行順序を変えても通ること。
- **一度 fail を確認する**: 期待値を意図的に壊して落ちることを確認してから戻す。

## 5. Mutation 耐性のあるアサーション

Mutation ツールがある場合は第一指標。無い場合も Level C レビューで同表を適用:

| 弱いアサーション               | 強いアサーション               |
| ------------------------------ | ------------------------------ |
| `toHaveBeenCalled()`           | `toHaveBeenCalledWith(具体値)` |
| `toBeTruthy()`                 | `toEqual({ ... })` / 完全な値  |
| `length > 0`                   | `toHaveLength(n)`              |
| エラーが「投げられること」のみ | メッセージ・コード・型まで検証 |
| 境界の片側のみ                 | 境界の両側（§1-②）             |

## 6. モックの規律

- モックしてよいのは**自分が所有しない境界**のみ: HTTP、DB、時計、ファイルシステム、外部 SDK。
- 自分のコード同士の呼び出しをモックしない。
- モックの応答は**実際の API が返す形**のフィクスチャを使う。

## 7. レイヤーごとの責務分担

| レイヤー                       | 守るもの                                   | 書く量     |
| ------------------------------ | ------------------------------------------ | ---------- |
| 単体（lib / hooks / services） | ロジック、境界値、エラー分岐               | 最多       |
| コンポーネント / ルート統合    | 配線: loading / error / empty / data、認可 | 代表ケース |
| E2E                            | クリティカルジャーニーの一筆書き           | 最少       |

**同じ観点を複数レイヤーで重複させない。**

## 8. アンチパターン

- カバレッジ目的だけの薄いテスト
- 巨大な beforeEach に全準備
- プライベート関数を export してテスト
- 落ちたテストを skip で塞ぐ
- テストのためだけの `isTest` 分岐
- **実装に合わせて期待値を調整する**
